# 010. Match routing

**Статус:** todo  
**Фаза:** dialog  
**Зависимости:** 007, 004, 005

## Описание

Определить маршрут матчинга по комбинации (пол пользователя, кого ищет). Три поддерживаемых сценария v1, четвёртый явно исключён.

## Scope

- Таблица маршрутизации:

| Пол | Ищу | Маршрут |
|-----|-----|---------|
| M | F | AI (нейросеть) |
| M | M | P2P (живой собеседник) |
| F | F | AI (нейросеть) |

- API: `POST /match/start` → `{ route: "ai" | "p2p", dialog_id? }`
- Валидация профиля перед матчем (зарегистрирован, нет активного диалога)
- Emit `queue.entered` с metadata `{ route, gender, seeking }`
- **F → M:** вернуть ошибку «сценарий недоступен» или предложить изменить анкету

## Acceptance criteria

- [ ] M+F всегда попадает в AI-ветку
- [ ] M+M попадает в P2P-очередь
- [ ] F+F всегда попадает в AI-ветку
- [ ] F+M → понятное сообщение пользователю, матч не начинается
- [ ] Повторный `start` при активном диалоге → отказ
- [ ] `match_route` пишется в `dialog.started` event

## Технические заметки

- Route resolver — чистая функция: `resolve_route(gender, seeking) -> MatchRoute`
- AI route: сразу создать dialog + назначить persona (задача 014, 029)
- P2P route: добавить в Redis queue (задача 021)
- Несовместимые пары P2P (M+M только если оба M и seeking M) — логика в 021

## Out of scope

- F → M (живой или AI) — не в v1
- Гео-фильтры, возрастные фильтры при матче
- Приоритетная очередь для premium
